# DevSync v1 Specification

## Objective

Build the initial `devsync` binary.

v1 focuses exclusively on:

* Git-aware workspace synchronization
* local ↔ remote development workflows
* safe bidirectional working tree sync
* remote-first development

v1 intentionally excludes:

* workspace migration
* node orchestration
* service orchestration
* machine lifecycle management
* distributed synchronization
* background daemons

---

# Primary Workflow

The developer primarily works remotely.

Occasionally they want to:

* pull a workspace locally
* develop locally temporarily
* synchronize changes safely

The goal is a single command:

```bash
devsync
```

run from anywhere inside a Git repository.

The command:

1. autodiscovers the repo
2. identifies the workspace
3. validates Git state
4. determines synchronization safety
5. synchronizes working tree changes using Mutagen

---

# Core Assumptions

## Remote-First

The remote machine is considered canonical most of the time.

Local workspaces are optional convenience environments.

---

## .git Is Never Synchronized

Mutagen excludes:

```text
.git
```

Git repositories remain independent.

DevSync coordinates them intentionally.

---

## Divergence Is Fatal

If:

* local and remote histories diverge
* branches mismatch
* synchronization becomes ambiguous

DevSync aborts.

No automatic merges.

No rebases.

No conflict resolution.

# Git Validation Model

This is the most important part of the system.

---

# Branch Validation

## Local Branch

```bash
git branch --show-current
```

## Remote Branch

Run remotely over SSH.

Branches must match.

If they do not match:

```text
ERROR: branch mismatch
local: feature/auth
remote: main
```

Abort.

---

# Commit State Validation

## Compare Histories

Use:

```bash
git rev-list --left-right --count local...remote
```

Possible outcomes:

---

## State: Equal

```text
0 0
```

Safe.

Proceed with synchronization.

---

## State: Local Ahead

```text
3 0
```

Behavior:

1. push local commits
2. update remote branch
3. run synchronization

---

## State: Remote Ahead

```text
0 2
```

Behavior:

1. pull remote commits locally
2. run synchronization

---

## State: Diverged

```text
2 4
```

Abort.

Output:

```text
Workspace histories diverged.
Resolve manually before syncing.
```

---

# Dirty Working Tree Policy

Initial recommendation:

* allow dirty trees
* warn clearly
* continue only if Git ancestry is safe

Potential future strict mode:

```bash
devsync --require-clean
```

---

# Synchronization Model

## Transport Layer

Use Mutagen.

DevSync orchestrates Mutagen sessions.

Mutagen handles:

* filesystem watching
* incremental synchronization
* reconnection
* performance
* transport reliability

---

# Synchronization Rules

## Bidirectional Working Tree Sync

Mutagen runs bidirectionally.

However:

* Git validation occurs first
* .git is excluded
* synchronization only occurs after safety checks pass

---

# Session Reuse

Mutagen sessions should persist.

DevSync should:

* discover existing sessions
* reuse sessions when possible
* avoid recreating sessions repeatedly

---

# Ignore Rules

node_modules

build

Potential future support:

* .devsyncignore

* framework presets

```text

```

---

# Remote Access Assumptions

v1 assumes:

* SSH access exists
* Tailscale connectivity exists
* remote hostnames already resolve

DevSync does not manage:

* VPNs
* networking
* machine wake/suspend
* port forwarding

---

# Error Handling Philosophy

DevSync should prefer:

* aborting safely
* explicit messaging
* transparency

rather than:

* guessing intent
* automatic repair
* hidden mutation

# Initial Non-Goals

Explicitly excluded from v1:

* automatic branch switching
* workspace migration
* multi-node replication
* daemon architecture
* tmux orchestration
* Docker orchestration
* service forwarding
* machine lifecycle APIs
* distributed state management
* synchronization conflict resolution

---

# Initial Success Criteria

v1 is successful if:

1. a user can run `devsync` from a local repo
2. the tool safely validates Git state
3. the tool synchronizes working tree changes safely
4. divergence is detected correctly
5. branch mismatches are detected correctly
6. synchronization is transparent and predictable
7. remote-first development becomes ergonomic

---

# Long-Term Direction

Future versions may evolve toward:

* workspace orchestration
* migratable workspaces
* node management
* service awareness
* remote lifecycle management
* distributed workspace mobility

However:

v1 intentionally remains focused on:

> safe Git-aware synchronization for remote-first development.
